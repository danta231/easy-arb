# 模块化架构地图 / Module Architecture Map

本文档是开发阶段的模块化索引。它回答三个问题：

1. 每个模块负责什么。
2. 每个模块拥有什么数据、输入和输出。
3. 每个模块的边界如何测试和验收。

中文说明：`25_Core_Architecture_Reference.md` 说明核心架构和 Rust 落地规则，`22_Development_Execution_Plan.md` 说明阶段计划。本文档补齐“模块级开发说明”，开发人员领取任务前应先确认本模块卡片。

使用 Codex 开发时，还必须阅读 `24_Codex_Development_Runbook.md`。该文档说明 Codex 如何按模块边界领取任务、验证和交接。

## 1. 模块分组

| 分组 | 模块 | 说明 |
|---|---|---|
| 基础层 | `arb-domain`, `arb-contracts`, `arb-config` | 提供领域基础类型、数据合同和只读配置。其他模块依赖它们，但它们不能依赖业务流程模块。 |
| 事实层 | `arb-eventstore`, `arb-ledger`, `arb-reconciliation` | 保存事件事实、账本事实和对账事实。事实只能追加，不能原地改写。 |
| 决策层 | `arb-strategy-api`, `arb-strategies`, `arb-risk` | 策略提出候选转换，风控决定是否允许继续。该层不能直接改变账户。 |
| 执行层 | `arb-execution`, `arb-venue-exec`, `arb-signing` | 把已批准决策转成执行计划、模拟执行或受控可变执行。真实签名默认关闭。 |
| 数据适配层 | `arb-venue-data` | 只读取外部行情、余额、仓位和场所健康状态，不能下单、撤单、转账或签名。 |
| 验证和运营层 | `arb-replay`, `arb-ops` | 回放、报告、事故查看和运营查询。默认只读，不承载核心业务规则。 |
| 装配层 | `arb-runtime` | 进程入口、任务编排和依赖装配。只能连接模块，不能放策略、风控、账本或执行规则。 |

## 2. 模块间事实流

核心事实流必须保持单向：

```text
外部场所
  -> arb-venue-data
  -> NormalizedEvent
  -> arb-eventstore
  -> PortfolioState
  -> arb-strategy-api / arb-strategies
  -> CandidatePortfolioTransition
  -> arb-risk
  -> RiskDecision
  -> arb-execution
  -> ExecutionPlan / ExecutionReport
  -> arb-ledger
  -> LedgerEntry
  -> arb-reconciliation
  -> Incident / ReconciliationResult
  -> arb-ops
```

中文说明：策略只能站在 `PortfolioState` 快照上提出候选转换。执行只能消费已经通过风控的 `RiskDecision`。账本只根据合同事实入账，不直接调用策略或执行适配器。

## 3. 数据产出和消费关系

| 数据对象 | 主要生产模块 | 主要消费模块 | 约束 |
|---|---|---|---|
| `Asset` | `arb-contracts` | 全部核心模块 | 资产标识必须稳定，不能使用展示名称作为主键。 |
| `Instrument` | `arb-contracts`, `arb-venue-data` | 策略、风控、执行、账本 | 工具属性必须能力化，不能按交易场所硬编码。 |
| `VenueCapability` | `arb-contracts`, `arb-config` | 策略、风控、适配器 | 能力不足时策略不能产生依赖该能力的候选转换。 |
| `NormalizedEvent` | `arb-venue-data` | `arb-eventstore`, `arb-replay` | 标准化事件必须包含来源、序号、新鲜度和原始事件引用。 |
| `PortfolioState` | 状态构建流程，归属 `arb-contracts` 合同 | 策略、风控、执行、对账 | 状态快照必须可哈希、可回放。 |
| `CandidatePortfolioTransition` | `arb-strategies` | `arb-risk` | 候选转换不是订单，不能包含签名结果或真实下单引用。 |
| `RiskDecision` | `arb-risk` | `arb-execution`, `arb-ops` | 每个决策必须含原因码、检查结果和策略版本。 |
| `ExecutionPlan` | `arb-execution` | 模拟执行、可变执行适配器、回放 | 执行计划必须来自已批准或需人工审批的风控决策。 |
| `ExecutionReport` | `arb-execution`, `arb-venue-exec` | `arb-ledger`, `arb-reconciliation`, `arb-ops` | 执行报告必须表达未知、部分成功、失败和补偿状态。 |
| `LedgerEntry` | `arb-ledger` | `arb-reconciliation`, `arb-ops` | 账本只能追加；修正必须用冲销或调整分录。 |
| `Incident` | `arb-reconciliation`, `arb-risk`, `arb-execution` | `arb-ops` | 事故必须关联触发事实、严重等级、处理状态和影响范围。 |

## 4. 模块卡片

### 4.1 `arb-domain`

- 定位：最底层领域基础模块。
- 输入：无内部输入，只接收调用方传入的原始字符串、十进制字符串和时间值。
- 输出：强类型 ID、金额、价格、利率、基点、时间、状态枚举、错误分类。
- 数据归属：领域基础类型定义和基础校验规则。
- 公开接口：构造函数、解析函数、格式化函数、基础比较和安全算术。
- 禁止行为：读取配置、访问网络、写事件、写账本、调用策略或执行。
- 测试责任：类型不能互相误用；十进制往返不丢精度；核心路径禁止 `f64`。
- 首次交付阶段：阶段 1。

### 4.2 `arb-contracts`

- 定位：所有 schema 对应 Rust 类型和合同校验模块。
- 输入：JSON、YAML 或调用方构造的合同对象。
- 输出：严格反序列化后的合同类型、规范序列化字节、合同校验错误。
- 数据归属：`Asset`、`Instrument`、`NormalizedEvent`、`PortfolioState`、`CandidatePortfolioTransition`、`RiskDecision`、`ExecutionPlan`、`ExecutionReport`、`LedgerEntry`、`Incident` 等合同类型。
- 公开接口：`from_json_strict`、`to_canonical_json`、`schema_version`、合同级 `validate`。
- 禁止行为：下单、签名、风控决策、账本入账、外部接口访问。
- 测试责任：每个 schema 有正例和反例；未知字段拒绝；枚举非法值拒绝；金额精度不丢失。
- 首次交付阶段：阶段 1。

### 4.3 `arb-config`

- 定位：只读配置、配置版本和安全开关模块。
- 输入：配置文件、环境覆盖值、策略版本引用、签名策略引用。
- 输出：已校验配置对象、配置哈希、执行模式、熔断状态、能力开关。
- 数据归属：配置结构、配置版本、配置哈希、默认安全策略。
- 公开接口：加载配置、校验配置、读取只读配置视图、计算配置哈希。
- 禁止行为：执行交易、保存明文密钥、直接改账户、把配置读取变成风控批准。
- 测试责任：默认实盘关闭；熔断开关生效；配置哈希稳定；非法配置拒绝启动。
- 首次交付阶段：阶段 1。

### 4.4 `arb-eventstore`

- 定位：追加式事件存储模块。
- 输入：标准化事件、系统内部事实事件、关联 ID。
- 输出：全局递增序号、事件哈希、事件读取流、关联事件链。
- 数据归属：事件顺序、事件哈希、事件索引、追加写入规则。
- 公开接口：追加事件、按序号读取、按关联 ID 查询、读取事件范围。
- 禁止行为：解释策略含义、修改历史事件、产生风控结论、直接写账本。
- 测试责任：事件只能追加；序号单调递增；哈希稳定；重放顺序一致。
- 首次交付阶段：阶段 2。

### 4.5 `arb-ledger`

- 定位：复式账本核心模块。
- 输入：执行报告、资金费用事件、费用事件、调整请求、冲销请求。
- 输出：账本分录、账户余额视图、分录哈希、冲销和调整记录。
- 数据归属：账本科目、借贷平衡规则、分录生命周期、余额视图。
- 公开接口：入账、冲销、调整、查询余额、查询分录链。
- 禁止行为：调用策略、调用执行适配器、读取密钥、原地修改历史分录。
- 测试责任：借贷必平；重复事件幂等；冲销不删除原分录；异常状态进入事故。
- 首次交付阶段：阶段 3。

### 4.6 `arb-reconciliation`

- 定位：对账和差异发现模块。
- 输入：账本视图、场所只读余额、场所只读仓位、成交回报、资本预留状态。
- 输出：对账结果、差异项、事故建议、运营摘要。
- 数据归属：对账规则、差异分类、容忍阈值、差异状态。
- 公开接口：运行对账、生成差异、读取未解决差异、生成对账摘要。
- 禁止行为：修正账本历史、直接下单补偿、绕过事故流程、调用签名。
- 测试责任：余额不一致能发现；仓位不一致能发现；已知容忍范围不误报；严重差异生成事故。
- 首次交付阶段：阶段 7。

### 4.7 `arb-strategy-api`

- 定位：策略只读接口和策略运行边界。
- 输入：组合状态快照、场所能力、只读配置、固定时间源。
- 输出：候选组合转换、策略拒绝原因、策略诊断信息。
- 数据归属：策略 trait、只读快照 trait、策略输出约束。
- 公开接口：读取快照、读取能力、读取时间、提交候选转换。
- 禁止行为：暴露下单、撤单、签名、转账、账本写入和运行时装配能力。
- 测试责任：策略接口无法访问执行模块；策略输出必须可序列化；策略时间源可固定。
- 首次交付阶段：阶段 4。

### 4.8 `arb-strategies`

- 定位：具体策略实现集合。
- 输入：`arb-strategy-api` 提供的只读快照和配置视图。
- 输出：`CandidatePortfolioTransition` 或明确拒绝原因。
- 数据归属：策略参数解释、机会识别逻辑、策略版本。
- 公开接口：注册策略、运行策略、返回候选转换。
- 禁止行为：依赖执行、签名、可变适配器、账本写入、运行时装配。
- 测试责任：样例策略在固定输入下输出稳定；能力不足时拒绝；非法依赖检查失败。
- 首次交付阶段：阶段 4。

### 4.9 `arb-risk`

- 定位：风控决策模块。
- 输入：候选组合转换、组合状态、场所能力、只读配置、风控策略版本。
- 输出：`RiskDecision`，包含批准、拒绝、需人工审批、原因码和检查明细。
- 数据归属：风控检查规则、原因码、阈值解释、风控策略版本。
- 公开接口：评估候选转换、读取检查结果、生成拒绝报告。
- 禁止行为：调度执行、签名、写账本、直接改变账户、调用可变适配器。
- 测试责任：数据过期拒绝；流动性不足拒绝；资本预留不足拒绝；未知状态按风险处理。
- 首次交付阶段：阶段 5。

### 4.10 `arb-execution`

- 定位：执行计划、执行腿状态机、资本预留和模拟执行模块。
- 输入：已批准或需人工审批通过的 `RiskDecision`、组合状态、只读配置。
- 输出：`ExecutionPlan`、资本预留状态、模拟 `ExecutionReport`、执行状态事件。
- 数据归属：执行状态机、执行计划构建规则、资本预留状态机、模拟器。
- 公开接口：构建执行计划、预留资本、释放资本、推进执行状态、生成执行报告。
- 禁止行为：直接持有密钥、绕过风控、直接写实盘账本、直接读取策略内部状态。
- 测试责任：非法状态迁移失败；拒绝决策不能生成执行计划；部分成功和未知状态可表达。
- 首次交付阶段：阶段 6。

### 4.11 `arb-venue-data`

- 定位：外部场所只读数据适配模块。
- 输入：交易场所行情、余额、仓位、工具信息、健康状态、原始消息；行情可以来自 `REST` 快照和 `WSS` 即 WebSocket Secure 流式推送。
- 输出：原始事件引用、标准化事件、只读账户快照、场所健康状态。
- 数据归属：只读适配器 trait、`REST` 快照启动和补洞规则、`WSS` 流式序号和断线状态、外部错误映射、限频和新鲜度元数据。
- 公开接口：拉取行情、订阅行情、协调 `REST` 快照与 `WSS` 流式更新、读取余额、读取仓位、读取场所健康。
- 禁止行为：下单、撤单、转账、签名、依赖 `arb-venue-exec`。
- 测试责任：断线重连可表达；消息乱序和序号缺口可检测；`WSS` 未完成 `REST` 快照启动时必须 fail closed；只读模块不能引入签名依赖。
- 首次交付阶段：阶段 8。

### 4.12 `arb-venue-exec`

- 定位：可变执行适配器边界和未来实盘实现位置。
- 输入：执行计划中的执行腿、受控签名请求、执行模式配置。
- 输出：下单回报、撤单回报、转账回报、未知状态回报、外部错误映射。
- 数据归属：可变执行 trait、场所提交结果、幂等键处理、外部订单引用。
- 公开接口：提交订单、撤销订单、查询可变动作状态、提交转账请求。
- 禁止行为：做策略判断、做风控批准、保存明文密钥、绕过 `arb-signing`。
- 测试责任：默认功能不开启时不参与编译；幂等键重复提交不会产生重复动作；未知状态进入对账。
- 首次交付阶段：阶段 11。

### 4.13 `arb-signing`

- 定位：签名和密钥访问边界。
- 输入：签名请求、签名策略引用、硬件或托管签名器返回值。
- 输出：签名结果、签名审计引用、签名失败原因。
- 数据归属：签名接口、签名策略、签名审计元数据。
- 公开接口：创建签名请求、执行受控签名、验证签名策略。
- 禁止行为：向日志、事件、报告输出明文密钥；被策略和运营工具直接依赖。
- 测试责任：默认无真实签名实现；日志脱敏；签名失败不被当作成功。
- 首次交付阶段：阶段 11。

### 4.14 `arb-replay`

- 定位：确定性回放和黄金结果比较模块。
- 输入：事件 fixture、配置 fixture、策略版本、风控版本、固定时间源。
- 输出：回放结果、差异报告、黄金结果比较状态。
- 数据归属：回放加载器、固定时间源、固定随机种子、比较规则。
- 公开接口：加载回放、运行回放、比较预期结果、输出差异。
- 禁止行为：访问外部接口、触发真实签名、写真实账户、依赖实盘适配器。
- 测试责任：同一输入多次运行结果一致；时间不依赖系统当前时间；差异报告可定位。
- 首次交付阶段：阶段 2，阶段 9 扩展端到端回放。

### 4.15 `arb-ops`

- 定位：命令行、日报、拒绝报告、事故报告和只读查询模块。
- 输入：事件、账本视图、对账结果、事故、风控拒绝记录。
- 输出：运营报告、日报、查询结果、人工审批材料。
- 数据归属：报告模板、查询命令、人工审批材料格式。
- 公开接口：生成日报、查询事故、查询拒绝原因、导出审查材料。
- 禁止行为：下单、签名、直接改变账户、修改账本历史。
- 测试责任：报告字段完整；敏感信息脱敏；只读命令不能触发可变动作。
- 首次交付阶段：阶段 7。

### 4.16 `arb-runtime`

- 定位：应用入口和任务装配模块。
- 输入：启动参数、已校验配置、模块实现、任务调度配置。
- 输出：运行中的进程、任务拓扑、健康状态、退出状态。
- 数据归属：启动流程、依赖注入、任务生命周期、优雅退出。
- 公开接口：启动服务、停止服务、装配任务、暴露健康检查。
- 禁止行为：写策略规则、写风控规则、写账本规则、写执行状态机规则。
- 测试责任：启动失败原因明确；熔断打开时不启动可变执行；任务退出可观测。
- 首次交付阶段：阶段 9。

## 5. 模块接口边界

以下接口名称是建议名，正式开发可微调，但职责不能合并。

| 接口 | 所属模块 | 调用方 | 说明 |
|---|---|---|---|
| `EventWriter` | `arb-eventstore` | 数据适配、执行、账本、事故流程 | 只追加事件，返回序号和哈希。 |
| `EventReader` | `arb-eventstore` | 回放、运营、状态构建 | 只读事件，不提供改写能力。 |
| `PortfolioSnapshotReader` | `arb-strategy-api` | 策略、风控 | 读取组合状态快照。 |
| `Strategy` | `arb-strategy-api` | `arb-strategies`, `arb-runtime` | 运行策略并返回候选转换。 |
| `RiskEvaluator` | `arb-risk` | `arb-runtime` | 评估候选转换并返回风控决策。 |
| `ExecutionPlanner` | `arb-execution` | `arb-runtime` | 根据风控决策生成执行计划。 |
| `ExecutionAdapter` | `arb-venue-exec` | `arb-execution` 或运行时装配层 | 提交可变执行动作，默认只用模拟实现。 |
| `SigningProvider` | `arb-signing` | `arb-venue-exec` | 提供受控签名，不暴露密钥。 |
| `LedgerWriter` | `arb-ledger` | `arb-runtime` | 根据合同事实追加账本分录。 |
| `ReconciliationRunner` | `arb-reconciliation` | `arb-runtime`, `arb-ops` | 运行对账并产生差异结果。 |

中文说明：接口不是为了提前抽象一切，而是为了防止跨边界调用。比如策略不应该能拿到 `ExecutionAdapter`，运营工具不应该能拿到真实 `SigningProvider`。

## 6. 模块和阶段对应关系

| 阶段 | 新增或重点模块 | 阶段退出时必须成立 |
|---|---|---|
| 阶段 0 | 全部空 crate、检查脚本 | 模块存在，非法依赖可被检查脚本发现。 |
| 阶段 1 | `arb-domain`, `arb-contracts`, `arb-config` | 基础类型、合同类型、配置边界可测试。 |
| 阶段 2 | `arb-eventstore`, `arb-replay` | 事件可追加、可哈希、可固定输入回放。 |
| 阶段 3 | `arb-ledger` | 复式账本可入账、冲销、调整并保持平衡。 |
| 阶段 4 | `arb-strategy-api`, `arb-strategies` | 策略只能读快照并输出候选转换。 |
| 阶段 5 | `arb-risk` | 风控能批准、拒绝或要求人工审批，并说明原因。 |
| 阶段 6 | `arb-execution` | 执行计划、资本预留、模拟执行状态机完成。 |
| 阶段 7 | `arb-reconciliation`, `arb-ops` | 对账独立运行，运营报告只读。 |
| 阶段 8 | `arb-venue-data` | 第一个只读场所适配器能输出标准化事件。 |
| 阶段 9 | `arb-runtime` | 只读数据到模拟执行、账本和对账的端到端链路跑通。 |
| 阶段 10 | `arb-ops`, `arb-execution` | 人工审批材料、审批记录和拒绝路径完整。 |
| 阶段 11 | `arb-venue-exec`, `arb-signing` | 可变执行和签名只完成边界与模拟实现，不接真实资金。 |
| 阶段 12 | 全部模块 | 通过受控实盘准备评审后，才允许讨论真实执行。 |

## 7. 模块完成定义

一个模块不能只因为能编译就算完成。每个模块至少满足以下完成定义：

1. 有清晰的公开接口，不暴露不属于本模块的能力。
2. 有正向测试、反向测试和边界测试。
3. 有依赖边界检查，不引入禁止依赖。
4. 有错误类型，错误不能只用字符串透传。
5. 有日志和事件字段脱敏要求。
6. 有固定输入下的确定性测试。
7. 有至少一个与 schema 或 fixture 对齐的样例。
8. 有中文说明，解释该模块的职责和限制。

## 8. 开发分工建议

| 工作流 | 推荐负责模块 | 分工理由 |
|---|---|---|
| 合同和类型工作流 | `arb-domain`, `arb-contracts`, `arb-config` | 先稳定数据结构，减少后续模块返工。 |
| 事实和回放工作流 | `arb-eventstore`, `arb-replay`, `arb-ledger` | 事件、回放和账本都依赖确定性事实。 |
| 策略和风控工作流 | `arb-strategy-api`, `arb-strategies`, `arb-risk` | 策略输出和风控输入必须紧密对齐，但不能共享执行能力。 |
| 执行安全工作流 | `arb-execution`, `arb-venue-exec`, `arb-signing` | 执行状态、可变动作和签名边界需要统一安全审查。 |
| 运营和对账工作流 | `arb-reconciliation`, `arb-ops` | 对账结果必须进入运营报告和事故处理。 |
| 装配工作流 | `arb-runtime` | 只在依赖稳定后装配，避免业务规则混入入口模块。 |

中文说明：如果团队人数少，也应按工作流顺序开发，而不是把策略、执行、适配器和账本写进同一个模块。模块边界比短期开发速度更重要。
