# 核心架构参考 / Core Architecture Reference

本文档是裁剪后的架构参考文档，用来替代早期分散的总览、领域模型、场所模型、策略、风控、执行、账本、日志、安全和验收说明。正式开发时，Codex 优先读取 `24_Codex_Development_Runbook.md`、`22_Development_Execution_Plan.md` 和 `23_Module_Architecture_Map.md`；当需要理解架构背景和不可变规则时，再读取本文档。

中文说明：本文档保留开发必须知道的架构规则，删除重复叙述。它不是阶段计划，也不是模块任务清单。

## 1. 核心定位

本系统不是单一套利 bot，而是通用组合状态转换平台。任何策略，无论是现货套利、永续价差、资金费率、期现、期权波动率、借贷利差、稳定币脱锚、桥延迟还是询价套利，都必须表达为当前组合状态到候选组合状态的转换。

不可变核心包含：

- 统一领域模型：资产、场所、工具、账户、余额、仓位、报价、订单簿、路线、腿、候选转换、风控决策、执行计划、执行报告、成交、账本分录、事故。
- 能力化场所模型：场所按能力声明，不按中心化或去中心化粗分类写死。
- 策略插件合同：策略只能读取标准化状态并输出候选组合转换。
- 统一执行管线：信号、候选转换、风控、资本预留、审批、执行模式、执行计划、执行报告、账本、对账、报告。
- 记录系统：无前端时，事件、日志、报告、账本和回放就是操作员界面。

## 2. 不可变原则

1. 第一天就采用最终架构。可以只运行一个适配器或一个策略，但核心边界不能临时简化。
2. 策略永远不拥有执行权限。策略不能下单、签名、转账、改保证金或写账本。
3. 执行模式是权限策略，不是架构分支。只读、模拟、人工审批、受控实盘和自动实盘使用同一条管线。
4. 所有关键事实必须事件化。没有事件记录，就不能视为系统事实。
5. 账本是权威经济事实。策略自己计算的盈亏不能替代账本。
6. 对账是强制流程，不是运营附属工具。
7. 语言是实现细节。Rust 实现必须服从 schema、状态机、事件和账本合同。
8. 没有前端也必须可运营。报告、结构化日志、事故记录和回放必须完整。

## 3. 统一领域模型

| 概念 | 说明 | 开发要求 |
|---|---|---|
| 资产 | BTC、ETH、USDC、包装资产、链原生 gas、抵押资产等经济价值单位 | 必须有稳定标识、精度、风险组和结算属性 |
| 场所 | 提供价格、流动性、托管、借贷、路由、撮合或结算能力的系统 | 必须按能力声明，不按名称硬编码逻辑 |
| 工具 | 可交易、可借贷、可报价或可结算对象 | 必须声明结算资产、保证金资产、tick、lot、乘数、到期、资金费率等属性 |
| 账户和托管位置 | 余额或仓位存在的位置，资产实际受控的位置 | 需要区分子账户、链上钱包、金库、托管账户和内部资金桶 |
| 余额和仓位 | 可用、锁定、预留、待结算、借入、借出、未实现盈亏等状态 | 必须可回放、可对账、可归因 |
| 候选转换 | 策略输出的组合状态变化建议 | 不是订单，不能包含签名或真实下单结果 |
| 执行计划 | 风控批准后生成的有序动作计划 | 必须包含依赖、超时、取消、对冲、补偿和失败语义 |
| 账本分录 | 权威经济记录 | 只能追加，修正必须用冲销或调整分录 |

Rust 实现要求：ID 使用 newtype，状态使用 enum，金额和价格使用定点十进制，核心路径禁止 `f64`。

## 4. 能力化场所模型

场所必须声明能力。适配器不能假设某个场所天然具备下单、撤单、私有流、链上结算或钱包签名。

能力维度：

- 市场能力：现货、保证金、永续、交割合约、期权、自动做市池、订单簿、询价、聚合路由、资金费率、借贷、预言机、跨桥。
- 执行能力：市价单、限价单、只挂单、只减仓、改单、批量订单、原子交换、链上交易、链下撮合、私有订单流、仅人工审批。
- 结算和托管能力：链上结算、链下托管、自托管钱包、子账户、组合保证金、逐仓、全仓、外部托管、跨桥结算。
- 认证能力：公开数据、API key、钱包签名、会话令牌、多签审批、硬件签名。
- 数据能力：轮询、WebSocket、历史 K 线、历史成交、资金费率历史、仓位历史、余额历史、限频头、序号。

中文说明：`arb-venue-data` 只读取数据，`arb-venue-exec` 才允许表达可变账户动作。两者不能合并。

## 5. 统一工具模型

核心工具类型包括：

- 现货交易对
- 保证金现货交易对
- 永续合约
- 交割合约
- 期权
- 自动做市池
- 订单簿市场
- 资金费率流
- 借款利率市场
- 出借利率市场
- 询价路线
- 聚合器路线
- 跨桥路线
- 预言机价格源
- 合成指数
- 结构化仓位

每个工具至少要描述定价属性、结算资产、保证金资产、到期时间、资金费率计划、合约乘数、最小价格单位、最小数量单位、流动性模型、执行模型和失败模型。

## 6. 组合状态和候选转换

组合状态是跨场所、跨账户、跨托管位置的标准化视图，至少包括：

- 时间戳和数据新鲜度
- 资产余额
- 工具仓位
- 未完成订单和撤单
- 待确认链上交易
- 待处理转账和跨桥结算
- 资本预留
- 保证金和强平指标
- 资金费率和费用累计
- 已实现和未实现盈亏
- 缺失数据标记和置信度

候选组合转换必须包含策略标识、策略版本、输入事件引用、当前组合状态引用、执行腿、预期状态变化、预期收益、持有周期、资本需求、保证金影响、库存影响、流动性影响、资金费率影响、失败模式、风险标记和假设。

中文说明：持有周期是强制字段。即时套利和资金费率持仓使用同一架构，但风险和资本策略不同。

## 7. 策略合同

策略输入：

- 标准化市场状态
- 组合状态
- 场所健康状态
- 风控策略快照
- 策略配置快照
- 允许的历史特征窗口
- 注入的时间源

策略输出：

- `CandidatePortfolioTransition`
- 策略诊断事件
- 策略拒绝事件

策略禁止行为：

- 直接下单
- 直接签名
- 直接转账
- 直接改保证金
- 直接访问密钥
- 静默修改状态
- 在账本外计算权威盈亏
- 忽略风控决策

策略生命周期包括注册、禁用、只读评估、可模拟、可人工审批、可实盘、暂停和废弃。这些是治理状态，不是架构变化。

## 8. 风控模型

风控是主要安全边界。风控决策类型包括批准、带约束批准、拒绝、需要人工审批、需要更多数据、被熔断暂停。

交易前必须检查：

- 数据新鲜度
- 场所健康
- 限频状态
- 流动性
- 滑点
- 手续费和 gas
- 库存边界
- 余额充足性
- 资本预留可用性
- 保证金充足性
- 强平距离
- 资金费率不确定性
- 基差扩大风险
- 单腿执行风险
- 链拥堵
- nonce 或待处理交易冲突
- 未完成订单冲突
- 策略敞口限制
- 日亏损和回撤限制
- 相关性和集中度限制

交易后必须检查成交、仓位、余额、账本完整性、残余敞口、对冲完整性、卡住订单、卡住交易、盈亏归因和事故生成。

中文说明：每个 `RiskDecision` 必须记录策略版本、输入状态、检查结果、阈值、约束、原因码和策略版本。

## 9. 资本、保证金和托管

资本不是单一余额。核心状态需要区分：

- 可用余额
- 锁定余额
- 策略预留资本
- 待结算资金
- 借入资金
- 出借资金
- 保证金占用
- 链上待确认资金
- 跨桥待到账资金
- 外部托管资金

资本预留必须有状态机，至少覆盖请求、已预留、转入执行、释放、过期和对账不一致。任何资本预留都要有关联事件、策略、候选转换和账本引用。

## 10. 执行管线

唯一执行管线：

1. 信号接入
2. 候选机会创建
3. 交易前风控
4. 资本预留
5. 审批策略
6. 执行模式选择
7. 执行计划构建
8. 场所特定准备
9. 分发或模拟分发
10. 执行报告收集
11. 失败处理
12. 交易后风控
13. 账本入账
14. 对账
15. 报告和告警

执行模式：

- 只读：只记录会被考虑的机会，不允许账户变化。
- 模拟：生成模拟执行报告和模拟成交，写入模拟账本命名空间。
- 人工审批：生成执行计划，但需要人工批准，审批本身是审计事件。
- 受控实盘：仅在严格额度、熔断和外部监控下允许真实动作。
- 自动实盘：需要独立治理批准、审查、告警、对账和事故流程。

未知状态必须风险关键处理，不能当作成功继续交易。

## 11. 事件、账本和记录

事件类别至少包括原始市场数据、标准化市场数据、场所健康、策略信号、候选机会、策略拒绝、风控决策、资本预留、审批、执行计划、执行分发、执行报告、成交、转账、资金费率、费用、余额快照、仓位快照、账本分录、对账、事故和审计事件。

事件必须包含事件标识、事件类型、事件版本、业务时间、写入时间、来源、关联标识、因果标识、策略标识、场所标识、工具标识、组合状态引用、schema 版本、载荷和哈希。

账本原则：

- 账本只能追加。
- 修正是调整分录，不是重写。
- 分录要保留借贷语义、资产、账户、托管位置、策略归因、机会归因、执行归因、费用归因、资金费率归因和盈亏归因。

无前端运行时，结构化记录就是操作界面。日报、机会报告、拒绝报告、场所健康报告、风险限额报告、余额仓位对账报告、资金费率报告和事故报告都必须由结构化事实生成。

## 12. 可观测、事故和回放

关键指标包括市场数据延迟、数据陈旧度、WebSocket 断连、API 错误、限频使用率、链上 RPC 落后块数、gas 价格、机会数量、拒绝数量、风控拒绝、执行失败、成交不一致、对账差异、已实现盈亏、未实现盈亏、资本使用率和熔断状态。

事故必须记录严重等级、触发事实、影响范围、处理人、状态、时间线、关联事件、关联账本和复盘引用。

回放必须能在固定事件、固定配置、固定策略版本、固定风控版本和固定时间源下重建策略决策、风控决策、执行计划、账本结果和对账结果。回放期间禁止访问外部 API。

## 13. Rust 实施规则

Rust 是当前核心实现语言。开发必须遵守：

- schema 对应 Rust 结构体。
- 金额、数量、价格、利率和盈亏使用 decimal 或定点封装。
- ID 使用 newtype，例如 `VenueId`、`AssetId`、`StrategyId`、`EventId`。
- 生命周期和状态机使用 enum。
- 策略 API 只读，和执行、签名 crate 分离。
- 配置解析必须版本化。
- 所有服务通过 schema、事件和账本对齐，不允许语言私有事件格式。

中文说明：允许将来用 Python 做研究、TypeScript 做运营界面、Go 做边缘采集，但核心合同不能分叉。

## 14. 安全和治理

密钥不能进入代码、日志、事件、报告或问题描述。密钥包括 API key、私钥、助记词、钱包签名、会话令牌、口令和 webhook token。

权限级别：

- 公开数据
- 只读账户
- 模拟执行
- 人工审批执行
- 受控实盘执行
- 自动实盘执行

治理要求：

- CEX API key 不应有提现权限。
- 自动化钱包应使用资金受限的隔离热钱包。
- 可移动资金的能力必须有明确治理审批。
- 部署、配置变更、风控策略变更、策略启用、适配器启用、密钥轮换和人工覆盖都必须生成审计事件。
- 熔断开关需要覆盖全局、策略、场所、账户、工具、资产、链和执行模式。

个人自用治理画像：

- 如果系统只由所有者本人使用自己的小额资金，可以采用个人小额受控试运行治理画像；该画像不是外部审计，也不能用于管理他人资金、团队资金或商业服务。
- 个人画像必须默认停留在只读、模拟或人工手动执行；任何系统发起的账户变更都必须先完成 `review/personal_guarded_live_governance.md` 的最低门槛。
- 个人小额受控试运行仍必须满足无提现权限、隔离账户、小额资金上限、单笔限额、日亏损限额、每笔人工确认、kill switch、未知状态停机、执行后强制对账和事故记录。
- AI 辅助自审只能作为内部准备材料，不能记为外部独立审查通过，不能接触真实 API secret、私钥、助记词、session token 或 webhook token。
- 自动实盘不属于个人小额受控试运行画像；启用自动实盘前仍需要独立治理批准和更严格审查。

## 15. 验收清单

架构验收：

- 策略不能执行。
- 所有执行走同一管线。
- 所有机会都是候选组合转换。
- 场所按能力建模。
- 关键事实全部事件化。
- 账本只能追加。
- 对账是强制流程。
- 执行模式是权限策略，不是架构分支。

模块验收：

- 每个模块有清晰接口、输入、输出和数据归属。
- 禁止依赖进入自动化检查。
- 策略模块不能访问执行、签名、账本写入和运行时装配。
- 只读场所模块不能访问可变执行或签名。
- 运营模块默认只读。
- 运行时只做依赖装配。

schema 验收：

- schema 有 `$id`、标题、说明、必填字段和额外字段策略。
- decimal 使用字符串，禁止浮点。
- 状态、模式、原因和生命周期字段使用枚举。
- 有正向和反向样例。
- Rust JSON 往返测试通过。

实盘前说明：本文档和项目内 review 记录是内部结构化自评，不是外部审计。真实资金上线前仍需独立安全、风控、账本和执行审查。
